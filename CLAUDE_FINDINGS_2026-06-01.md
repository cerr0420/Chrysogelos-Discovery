In the `WALManager` review, I said combining the length prefix and payload into a single `write()` call means "the atomicity guarantee `O_APPEND` actually gives you... covers the whole record." That's not fully precise. `O_APPEND` guarantees the **seek-to-EOF-and-write is atomic with respect to positioning** — two concurrent `O_APPEND` writers won't have their writes overlap or clobber each other's offset. It does **not** by itself guarantee that a single `write()` call transfers all requested bytes as one indivisible unit — POSIX permits `write()` to return a short count on a regular file for reasons other than error (e.g., signal delivery being one, but also just resource limits on some implementations), which is exactly why the retry loop is needed. In practice on Linux with buffered I/O, a single `write()` of a few hundred bytes to a regular file essentially always completes fully in one call, but I described that as a guarantee when it's closer to "reliable in practice, defended by the retry loop as backstop" — the retry loop is doing real work, not just defensive boilerplate for a case that "can't happen."
